# Лабораторна робота №7

## Управління розробкою програмних засобів

### 1. Постановка задачі

Метою лабораторної роботи є аналіз методів декомпозиції, архітектурних принципів, стратегій тестування та критичності ПЗ у контексті реального проекту. Обрано 4 питання з переліку: об'єктна декомпозиція (20), зчеплення-зв'язність компонент (22), обсяги тестування (23), аналіз критичності ПЗ (11). Огляд побудовано на прикладі проекту «Track List».

---

### 2. Функціональна та об'єктна декомпозиція (питання 20)

#### Теоретичний огляд

Принцип «розділяй і володарюй» — фундамент проєктування ПЗ. Два основні підходи до декомпозиції:

**Функціональна декомпозиція** розкладає систему на функції (процедури), які трансформують вхідні дані у вихідні. Кожна функція — окрема операція. Підхід природний для потокової обробки даних, але погано масштабується при зростанні складності: функції починають ділити стан, виникають неявні залежності.

**Об'єктна декомпозиція** групує дані та поведінку в об'єкти. Три ключові механізми:

- **Інкапсуляція** — приховування внутрішнього стану за публічним інтерфейсом. Зміна реалізації не зачіпає код, що використовує об'єкт.
- **Наслідування** — побудова ієрархій для повторного використання коду та поліморфної поведінки.
- **Поліморфізм і пізнє зв'язування** — код працює з абстракцією, конкретна реалізація визначається в рантаймі.

#### Декомпозиція в Track List

Бекенд Track List побудовано на C#/.NET з тришаровою об'єктною декомпозицією:

```
Controllers → Services (через інтерфейси) → Repositories (через UnitOfWork)
```

**Шар контролерів** відповідає лише за HTTP-семантику: приймає запит, викликає метод сервісу, перетворює `Result<T>` у HTTP-відповідь. Наприклад, `ReviewController` інжектується лише з `IReviewService` — він не знає ні про Entity Framework, ні про структуру бази даних.

**Шар сервісів** містить бізнес-логіку. Кожен сервіс має інтерфейс (`IAuthService`, `IReviewService`, `IUserService` тощо — 9 інтерфейсів), що реалізує інкапсуляцію: контролер працює з абстракцією, конкретна реалізація підставляється через DI.

**Шар репозиторіїв** інкапсулює доступ до даних за патерном Unit of Work. 14 репозиторіїв агреговані в єдиний `IUnitOfWork` — сервіс отримує одну залежність замість чотирнадцяти.

Поліморфізм і пізнє зв'язування реалізовані через DI-контейнер .NET:

```csharp
builder.Services.AddScoped<IReviewService, ReviewService>();
builder.Services.AddScoped<IUnitOfWork, UnitOfWork>();
```

Контролер не знає конкретного типу сервісу — він працює з інтерфейсом. Це дозволяє замінити реалізацію (наприклад, для тестування) без зміни коду контролера.

#### Практичне спостереження

Об'єктна декомпозиція — не самоціль. У Track List наслідування використовується мінімально: сервіси не утворюють ієрархій, бо кожен епік має унікальну логіку. Замість глибоких ієрархій — плоска структура з інтерфейсами та композицією. Це свідомий вибір: наслідування заради наслідування створює крихкі залежності, де зміна базового класу ламає всіх нащадків.

### 3. Критерії зчеплення-зв'язності компонент (питання 22)

#### Теоретичний огляд

Два ключові критерії якості архітектури ПЗ:

**Зв'язність (Cohesion)** [1] — наскільки елементи всередині модуля пов'язані між собою. Висока зв'язність означає, що модуль відповідає за одну чітку задачу. Шкала (від найгіршої до найкращої): випадкова → логічна → часова → процедурна → комунікаційна → послідовна → функціональна.

**Зчеплення (Coupling)** [1] — наскільки модулі залежать один від одного. Низьке зчеплення означає, що зміна одного модуля не вимагає зміни інших. Шкала (від найгіршої до найкращої): за змістом → за спільними даними → за зовнішніми даними → за керуванням → за структурою даних → за даними → за повідомленнями.

**Факторизація** — процес виділення спільної функціональності в окремі модулі для усунення дублювання та зниження зчеплення.

Ідеал: **висока зв'язність + низьке зчеплення** — кожен модуль робить одну справу і мінімально залежить від інших.

#### Зчеплення-зв'язність в Track List

**Приклад високої зв'язності.** `ReviewService` містить всю логіку рецензій: створення, редагування, видалення, лайки, коментарі. Все, що стосується рецензій — в одному місці. Зміна бізнес-правил рецензій не зачіпає `AuthService` чи `UserService`.

**Приклад контрольованого зчеплення.** `ReviewService` залежить від `IUnitOfWork`, через який звертається до `ReviewRepository`, `UserRepository`, `CommentLikeRepository`. Це усвідомлений компроміс: сервіс рецензій потребує доступу до даних користувачів (для перевірки існування автора) та лайків (для підрахунку). Альтернатива — інжектити `IUserService` у `IReviewService` — створила б циклічну залежність між сервісами. UnitOfWork розв'язує цю проблему, даючи доступ до даних без залежності від чужої бізнес-логіки.

**Result як контракт між шарами.** Патерн `Result<T>` — приклад зчеплення за даними (один із найнижчих рівнів): шари обмінюються типізованим об'єктом з двома станами (success/failure). Репозиторій повертає `Result<T>`, сервіс перевіряє `IsFailure` і або обробляє помилку, або передає далі. Контролер перетворює `Result` у HTTP-код. Жоден шар не використовує виняткових ситуацій (exceptions) як контроль потоку — помилка є частиною типу, а не побічним ефектом.

```csharp
// Контролер — зчеплення лише за Result<T>
var result = await _reviewService.CreateAsync(dto);
if (result.IsFailure)
    return BadRequest(result.Error);
return Ok(result.Value);
```

**Важливе обмеження Result-патерну.** `Result<T>` працює лише за умови суворого дотримання неймінгових конвенцій та продуманих назв методів. Кожне повідомлення про помилку в Result пишеться розробником вручну — `Result.Fail("User not found")`, `Result.Fail("Review already exists")` — що створює значну прозорість під час дебагу: коли щось падає, текст помилки одразу вказує на причину та місце в бізнес-логіці. Але ця прозорість тримається виключно на дисципліні: якщо метод має неочевидну назву або повідомлення про помилку написано ліниво (`Result.Fail("error")`), патерн перетворюється з інструменту ясності в джерело плутанини. У Track List це працює, бо всі Result-повідомлення проходять code review.

#### Антипатерн: over-engineering абстракцій

Прагнення до мінімального зчеплення може стати самоціллю. Наприклад, патерн «один інтерфейс — одна реалізація» (як у Track List: `IReviewService` → `ReviewService`) часто критикується як зайвий рівень абстракції, якщо заміна реалізації не передбачається. У проекті Track List це виправдано тестуванням (моки через інтерфейси), але в менших проектах — це церемоніальний код без практичної користі. Три рядки конкретного коду краще за один рядок з абстракцією, яка ніколи не буде замінена.

### 4. Визначення обсягів інтеграційного та системного тестування (питання 23)

#### Теоретичний огляд

Тестування ПЗ організоване в піраміду (від основи до вершини):

1. **Юніт-тести** — перевірка окремих методів/класів у ізоляції. Швидкі, дешеві, але не виявляють проблем інтеграції.
2. **Інтеграційні тести** — перевірка взаємодії між компонентами (сервіс + база даних, API + зовнішній сервіс). Повільніші, але виявляють реальні проблеми на стиках.
3. **Системні тести** — перевірка всієї системи як чорної скриньки. Найдорожчі, але перевіряють реальну поведінку.

Визначення обсягу тестування — балансування між вартістю тестів та ризиком пропущених дефектів.

#### Стратегія тестування в Track List

У Track List тестування організоване прагматично, а не за підручником:

Таблиця: Стратегія тестування в Track List

| Рівень               | Реалізація                                             | Обсяг                                  |
| -------------------- | ------------------------------------------------------ | -------------------------------------- |
| BDD-тести (SpecFlow) | Сценарії user stories у Gherkin → автоматизовані тести | Покривають основний шлях кожного епіка |
| Performance-тести    | Навантажувальне тестування ключових ендпоінтів         | Авторизація, CRUD операції             |
| Code review          | Ручний огляд кожного PR перед злиттям                  | 100% коду проходить ревью              |
| Ручне тестування     | Перевірка FE після інтеграції з BE                     | Кожна нова сторінка                    |

**Що не тестується і чому:**

- **Юніт-тести окремих репозиторіїв** — репозиторії є тонкою обгорткою над Entity Framework. Тестувати їх окремо означає тестувати ORM, а не бізнес-логіку.
- **Edge cases конфігурації** — незареєстрований сервіс у DI-контейнері (реальний баг у Track List) не виявляється юніт-тестами, бо тести мокають залежності. Такі помилки виявляє перший запуск застосунку.
- **CSS/верстка** — візуальні регресії не покриваються автоматизованими тестами. Виявляються при ручному тестуванні.

#### 100% покриття — економічно недоцільний міф

Покриття тестами — одна з метрик, яку часто подають як ціль. На практиці:’

- **80% покриття** досягається за 20% зусиль (основні сценарії, happy path).
- **Наступні 15%** (edge cases, граничні значення) вимагають стільки ж зусиль, скільки перші 80%.
- **Останні 5%** (конфігурація, інфраструктура, race conditions) часто неможливо протестувати юніт-тестами в цілому.

BDD-підхід у Track List обирає інший фокус: замість покриття рядків коду — покриття user stories. Якщо всі сценарії з feature-файлів проходять — застосунок працює з точки зору користувача. Решту ловить code review та ручне тестування.

### 5. Аналіз критичності ПЗ (питання 11)

#### Теоретичний огляд

**Критичне ПЗ** — програмне забезпечення, приховані дефекти якого можуть призвести до необоротних наслідків: загрози життю, значних фінансових втрат, екологічних катастроф. Системи реального часу (авіоніка, медичне обладнання, АЕС, транспорт) є класичним контекстом критичного ПЗ.

Аналіз критичності включає:

- **Ідентифікацію прихованих дефектів** — дефектів, які не проявляються при нормальній роботі, але активуються при специфічних умовах.
- **Оцінку ризиків** — ймовірність активації дефекту × потенційний збиток.
- **Функціональну безпеку (safety)** — властивість системи перебувати в стані допустимих ризиків.
- **Інформаційну безпеку (security)** — захист від несанкціонованого доступу та маніпуляцій.

Для критичного ПЗ застосовуються формальні методи верифікації [5]: model-checking, темпоральна логіка, статичний аналіз, незалежна верифікація з диверсифікацією методів.

#### Therac-25: коли критичність — не абстракція

Найвідоміший випадок, що демонструє реальну ціну дефектів у критичному ПЗ — інциденти з радіотерапевтичним апаратом **Therac-25** (1985–1987) [2]. Через race condition у керуючому ПЗ апарат доставляв пацієнтам дозу радіації, що в сотні разів перевищувала терапевтичну. Шість відомих інцидентів, три з яких закінчились смертю пацієнтів.

Причини:

- **Відсутність апаратних блокувань** — попередня модель Therac-20 мала апаратні запобіжники, які компенсували баги ПЗ. У Therac-25 їх прибрали, повністю поклавшись на програмний контроль.
- **Race condition** — оператор міг змінити режим роботи швидше, ніж ПЗ обробляло попередню команду. Апарат переходив у режим високої енергії з параметрами низькоенергетичного режиму.
- **Повторне використання коду** без повторної верифікації — код від Therac-20 перенесли «як є», не враховуючи відсутність апаратних запобіжників.
- **Відсутність незалежної верифікації** — тестування проводила та ж команда, що писала код.

Therac-25 — це не історичний курйоз, а демонстрація того, навіщо існують формальні методи верифікації критичного ПЗ. Коли помилка в коді може вбити — model-checking, незалежна верифікація та кваліфікаційні випробування дійсно виправдані на всі 100%.

#### Спектр критичності: не все ПЗ — ядерний реактор

Проблема виникає, коли практики критичного ПЗ механічно переносяться на некритичні системи. Спектр критичності:

Таблиця: Спектр критичності ПЗ

| Рівень   | Приклад                                      | Ціна дефекту             | Методи верифікації                                         |
| -------- | -------------------------------------------- | ------------------------ | ---------------------------------------------------------- |
| Критичне | Авіоніка, медичне обладнання, АЕС            | Людські життя            | Формальна верифікація, model-checking, незалежний аудит    |
| Високе   | Банківські системи, платіжні шлюзи           | Значні фінансові втрати  | Обов'язкова сертифікація (PCI DSS), penetration testing    |
| Середнє  | Великі SaaS-платформи, хмарна інфраструктура | Втрата доходу, репутації | Автоматизовані тести, CI/CD, моніторинг, incident response |
| Низьке   | Веб-застосунки, MVP, дипломні проекти        | Незручність користувача  | Code review, BDD-тести, ручне тестування                   |

Track List — нижній рівень спектру. Якщо пошук фільмів поверне помилку — користувач оновить сторінку. Якщо авторизація зламається — команда виправить протягом дня.

#### GitHub: коли «середнє» претендує на «високе»

Показовий приклад зі спектру критичності — **GitHub**. Платформа, від якої залежать мільйони розробників та CI/CD-пайплайни по всьому світу, де зупинка сервісу буквально зупиняє розробку. За індустріальними стандартами, платформа такого масштабу мала б забезпечувати три дев'ятки доступності (99,9% — не більше 8,7 годин простою на рік).

На практиці GitHub регулярно має інциденти, що виходять за межі цього показника. Сторінка githubstatus.com [3] фіксує десятки інцідентів щомісяця різного ступеня серйозності. У 2024 та 2025 роках платформа мала кілька масштабних збоїв, що тривали годинами та впливали на GitHub Actions, Pull Requests та API.

Це не означає, що GitHub — поганий продукт. Це означає, що навіть компанія з необмеженими ресурсами (Microsoft) не може гарантувати три дев'ятки для складної розподіленої системи. Теорія вважає 99,9% базовим показником — практика показує, що для розподілених систем це недосяжний ідеал.

#### Visa: коли дешевше не виправляти

Ще один показовий кейс — відома вразливість у платіжній системі **Visa**, пов'язана з MIM-атакою (Man-in-the-Middle) на протокол EMV [4]. Зловмисник, перехоплюючи комунікацію між карткою та терміналом, може модифікувати дані транзакції: змінити тип термінала на «транзитний» (transit terminal) та позначити суму як «низьковартісну» (low-value transaction). При цих умовах протокол дозволяє транзакцію без введення PIN-коду та без обмеження суми — фактично, необмежене списання з картки жертви.

Вразливість відома дослідникам безпеки та публічно задокументована. Позиція Visa: виправлення вразливості вимагає зміни протоколу на рівні всієї глобальної інфраструктури (мільйони терміналів, банки-емітенти, процесингові центри). Прес-департамент Visa наголошує, що **дешевше компенсувати постраждалим клієнтам кошти за шахрайські транзакції, ніж змінювати систему**.

З позиції інженерної етики складно погодитись із рішенням свідомо зберігати відому вразливість, що безпосередньо загрожує фінансовій безпеці користувачів. Водночас, з точки зору аналізу критичності, логіка цього рішення зрозуміла: навіть у фінансовій сфері (рівень «Високе» у спектрі критичності) рішення про виправлення дефекту приймається на основі економічного розрахунку, а не формальних вимог безпеки.

#### Критичність та Track List

Аналіз критичності Track List:

Таблиця: Аналіз критичності Track List

| Характеристика                  | Оцінка                                                                     |
| ------------------------------- | -------------------------------------------------------------------------- |
| Функціональна безпека (safety)  | Не застосовується — застосунок не керує фізичними процесами                |
| Інформаційна безпека (security) | Середня — є персональні дані (email, пароль), захист через JWT + хешування |
| Приховані дефекти               | Виявляються при використанні, виправляються в наступному PR                |
| Ціна простою                    | Мінімальна — застосунок не є критичною інфраструктурою                     |

Єдиний аспект критичності, релевантний для Track List — **інформаційна безпека**: хешування паролів, JWT-автентифікація, RBAC-політики. Це не формальна верифікація, а базова гігієна веб-розробки. Code review виявив витік токенів у localStorage (фронтенд зберігав JWT одночасно в httpOnly cookies і localStorage, нівелюючи захист від XSS) — проблему, яку model-checking не виявив би в принципі, бо вона стосується архітектурного рішення, а не логіки виконання.

### 6. Висновки

Чотири розглянуті аспекти утворюють послідовність від конкретного до абстрактного: як декомпозувати код (ООП), як оцінити якість декомпозиції (зчеплення-зв'язність), як перевірити результат (тестування), і коли перевірка має бути формальною (критичність).

Ключовий висновок: рівень формалізації має відповідати рівню критичності. Therac-25 показав, що для медичного ПЗ формальна верифікація — не бюрократія, а необхідність: ціна дефекту — людське життя. GitHub показав, що навіть необмежені ресурси не гарантують ідеальної доступності складних систем. Visa показала, що навіть у фінансовій сфері рішення про виправлення відомої вразливості приймається на основі економіки, а не формальних вимог. Застосовувати практики авіоніки до веб-застосунку — такий самий інженерний прорахунок, як не застосовувати їх до медичного обладнання.
