### 4. Розрахункові формули

**Прямий прохід (ранні терміни):**

- Ранній початок (ES): ES = max(EF усіх попередників). Для робіт без попередників ES = 0.
- Раннє закінчення (EF): EF = ES + тривалість

**Зворотний прохід (пізні терміни):**

- Пізнє закінчення (LF): LF = min(LS усіх наступників). Для робіт без наступників LF = тривалість проекту.
- Пізній початок (LS): LS = LF − тривалість

**Резерв часу:**

- Резерв = LS − ES = LF − EF

Роботи з нульовим резервом лежать на **критичному шляху**.

### 6. Критичний шлях

**Критичний шлях:** BE-001 → BE-002 → BE-101 → BE-201 → BE-701 → FE-701 → QA-001

**Тривалість критичного шляху:** 6 + 5 + 10 + 6 + 5 + 6 + 10 = **48 людино-годин**

Інтерпретація: критичний шлях проходить через ініціалізацію проекту, автентифікацію, профілі, адміністрування та завершується ручним тестуванням. Затримка будь-якої з цих робіт напряму збільшує загальну тривалість проекту.

Роботи, що лежать поза критичним шляхом, мають резерви часу. Найбільші резерви мають роботи, які не мають залежностей та не є попередниками QA-001: FE-004 (46 год), FE-602 (45 год), BE-005 (45 год), BE-004 (44 год).

### 7. Висновки

У ході виконання лабораторної роботи побудовано мережевий графік проекту «Track List» з 51 роботою та визначено критичний шлях:

**BE-001 → BE-002 → BE-101 → BE-201 → BE-701 → FE-701 → QA-001** (48 год)

Виконано прямий та зворотний прохід мережевого графіка. Для кожної роботи розраховано ранній початок (ES), раннє закінчення (EF), пізній початок (LS), пізнє закінчення (LF) та резерв часу. 7 робіт мають нульовий резерв і є критичними. Роботи з великим резервом (інфраструктурні: Docker, Serilog, Error Pages) можуть бути відкладені без впливу на загальний термін проекту.

Критичний шлях логічно відповідає ланцюжку залежностей Backend → Frontend → QA, що підтверджує waterfall-характер обраної методології розробки.
