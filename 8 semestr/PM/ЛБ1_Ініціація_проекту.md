# Лабораторна робота №1

## Група процесів ініціації проекту

### 1. Постановка задачі

Метою лабораторної роботи є отримання практичних навичок ініціації програмного проекту: визначення обмежень, складання статуту проекту, технічного завдання та матриці відповідальності.

Об'єктом дослідження є проект **«Diploma Track List»** — веб-сервіс для комплексного управління користувацьким досвідом взаємодії з мультимедійним контентом (фільми, серіали, ігри, книги).

---

### 2. Опис проекту

**Diploma Track List** — це централізована веб-платформа, що вирішує проблему децентралізації трекінгу мультимедійного контенту. Сьогодні користувачі використовують різні сервіси для відстеження фільмів (Letterboxd), ігор (HowLongToBeat), книг (Goodreads) тощо. Проект об'єднує ці функції в єдиному рішенні.

**Основні функції:**

- централізоване керування колекціями медіа-контенту зі статусами («Переглянуто», «У планах», «Покинуто»);
- відстеження прогресу (переглянуті епізоди, прочитані сторінки, ігровий час);
- соціальна взаємодія: рецензії, оцінки, коментарі, підписки на інших користувачів;
- стрічка активності (персональна та глобальна);
- пошук з інтеграцією зовнішніх API (TMDB) та кешуванням на вимогу;
- адміністрування та модерація контенту.

**Стек технологій:**

- Backend: C#, .NET 10, Entity Framework Core, PostgreSQL;
- Frontend: TypeScript, SvelteKit, TailwindCSS v4;
- Інфраструктура: Docker, Nginx, Serilog;
- Безпека: JWT, хешування паролів, RBAC (User/Moderator/Admin);
- Тестування: XUnit, SpecFlow (BDD), Cucumber.js.

#### ПРИЧИНИ ІНІЦІАЦІЇ ПРОЕКТУ

Існуючі рішення для трекінгу мультимедійного контенту фрагментовані: користувач змушений використовувати окремі сервіси для фільмів, ігор та книг. Жодна платформа не пропонує єдиного уніфікованого рішення з повним набором соціальних функцій. Проект ініціюється як дипломна робота для демонстрації навичок повного циклу розробки веб-застосунку.

#### ЗМІСТ ПРОЕКТУ

Проект включає:

- розробку REST API (Backend) з 10 функціональних епіків;
- розробку клієнтського веб-застосунку (Frontend);
- інтеграцію із зовнішнім API (TMDB) для отримання метаданих медіа-контенту;
- налаштування інфраструктури (Docker, CI/CD, Nginx);
- написання автоматизованих тестів (BDD, unit, performance).

Проект **не** включає:

- розробку мобільного застосунку;
- підтримку інших зовнішніх API окрім TMDB (але архітектура застосунку дозволяє просте розширення списку інтеграцій);
- монетизацію та комерційний запуск.

#### ЦІЛІ ПРОЕКТУ

1. Створити функціонально повний веб-застосунок для трекінгу мультимедійного контенту.
2. Продемонструвати навички командної розробки з використанням сучасного стеку технологій.
3. Реалізувати всі епіків відповідно до специфікацій у feature-файлах.

#### РЕЗУЛЬТАТ ПРОЕКТУ

Працюючий веб-сервіс з повним функціоналом: реєстрація/авторизація, профілі, трекінг, рецензії, колекції, модерація, адміністрування, пошук та стрічка активності. Розгортання на сервері — у місяць захисту.

#### ОБМЕЖЕННЯ ТА ДОПУЩЕННЯ

Обмеження наведені у розділі 3. 

Допущення:

- зовнішній API TMDB залишатиметься доступним та безкоштовним протягом усього періоду розробки;
- команда зможе виділяти достатньо часу попри навчання та роботу;
- буде доступний сервер для розгортання у місяць захисту.

#### 5.1. Загальні відомості

**Назва продукту:** Track List

#### 5.3. Бізнес-правила (BRL)

- **BRL-1:** Гість може переглядати публічний контент, але не може виконувати дії (оцінити, підписатись, додати в список).
- **BRL-2:** Пароль має містити мінімум 8 символів, літери та цифри.
- **BRL-3:** Кожен медіа-елемент може мати лише один статус від одного користувача.
- **BRL-4:** Один користувач може залишити лише одну рецензію на один медіа-елемент.

#### 5.4. Ролі користувачів

Таблиця: Ролі користувачів

| Роль          | Права                                                              |
| ------------- | ------------------------------------------------------------------ |
| Гість         | Перегляд публічного контенту                                       |
| Користувач    | Усі дії з контентом: рецензії, оцінки, трекінг, колекції, підписки |
| Модератор     | + модерація скарг, перекладів                                      |
| Адміністратор | + управління ролями, користувачами, статистика, експорт            |

#### 5.5. Вимоги до безпеки

- Автентифікація через JWT токени (stateless);
- RBAC-політики (userPolicy, moderatorPolicy, adminPolicy);
- Хешування паролів;
- Валідація вхідних даних (FluentValidation);
- Захист від системних нікнеймів при реєстрації (admin, moderator тощо);
- Soft Delete замість фізичного видалення даних;
- Global Exception Handler (без стек-трейсів у продакшн-відповідях).

#### 5.6. Вимоги до інфраструктури

- Контейнеризація через Docker (docker-compose: API + Frontend + PostgreSQL + Nginx);
- Branching strategy: feature branches → code review → merge to main;
- Структурне логування (Serilog) у файл та консоль.

### 7. Висновки

У ході виконання лабораторної роботи було проведено ініціацію проекту «Track List». Визначено обмеження проекту (часові, бюджетні, ресурсні, технологічні, кваліфікаційні, законодавчі та якісні). Складено статут проекту, що фіксує мету, зміст, результати та обмеження. Розроблено технічне завдання з описом 9 функціональних епіків, бізнес-правил, ролей користувачів та вимог до безпеки та інфраструктури. Побудовано матрицю відповідальних, що розподіляє роботу між трьома членами команди: Team Lead/DevOps (Астаф'єв), Frontend (Петренко), Backend (Пилипчук).

Проект використовує waterfall-підхід зі спринтами як тайм-боксами: 9 спринтів (Sprint 0–8), кожен з яких реалізує окрему функціональну фазу. Координація між Frontend та Backend здійснюється через OpenAPI-контракт та Code Review при злитті feature-гілок у main.
