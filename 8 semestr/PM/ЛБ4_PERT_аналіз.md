# Лабораторна робота №4

## Аналіз термінів завершення проекту методом Resource-Leveled Scheduling

### 1. Постановка задачі

Метою лабораторної роботи є проведення аналізу термінів завершення проекту «Track List» з урахуванням реальних обмежень команди. За основу взято критичний шлях з ЛБ3 та оцінки трудомісткості (Н-оцінки) з ЛБ2.

Класичні методи мережевого планування (CPM) припускають необмежену кількість виконавців — усі паралельні гілки графіка виконуються одночасно. Для команди з 3 людей це припущення не працює. Resource-Leveled Scheduling (RLS) вирішує цю проблему, відштовхуючись від реальної доступності людей.

---

### 2. Критичний шлях проекту (з ЛБ3)

**BE-001 → BE-002 → BE-101 → BE-201 → BE-701 → FE-701 → QA-001**

Мережевий графік з ранніми/пізніми термінами та резервами наведено в ЛБ3.

Детерміністичний термін по критичному шляху (CPM): **48 людино-годин**. Цей термін передбачає, що 5 паралельних гілок після BE-001 справді виконуються одночасно. У реальності їх усі робить одна людина — Backend-розробник, на якому висить 21 задача (~138 людино-годин).

---

### 3. Метод Resource-Leveled Scheduling

Resource leveling (вирівнювання ресурсів) — це метод планування, який відштовхується не від залежностей між задачами, а від реальної доступності людей.

Якщо у вас один бекендер, то всі Backend-задачі — це одна черга, незалежно від того, що мережевий графік показує їх як паралельні. Метод «розтягує» графік, зсуваючи паралельні задачі в послідовні, доки кожен виконавець має не більше однієї задачі в кожен момент часу.

Коли у вас в команді 3 людини, майже ніколи не буває ситуації, де всі задачі йдуть паралельно — хтось завжди чекає на когось іншого.

Порахуємо навантаження кожного члена команди (Н-оцінки з ЛБ2):

**Петренко (Frontend):** FE-001(6) + FE-002(8) + FE-101(8) + FE-102(4) + FE-901(6) + FE-902(6) + FE-201(6) + FE-202(2) + FE-203(4) + FE-401(8) + FE-402(6) + FE-501(3) + FE-801(5) + FE-802(5) + FE-803(6) + FE-804(3) + FE-601(3) + FE-602(3) + FE-603(5) + FE-604(5) + FE-701(6) + FE-702(10) + FE-004(2) = **134 год (формально)**

Однак Петренко — початківець у фронтенд-розробці (перший досвід зі SvelteKit і Tailwind). На практиці час виконання задач для junior-розробника було б варто множити мінімум на 1,5×. Реалістична оцінка: **~200 год**.

**Астаф'єв (TL/DevOps/QA):** BE-003(6) + FE-003(5) + DO-001(6) + BE-004(4) + BE-005(3) + QA-001(10) + QA-002(8) = **42 год (формально)**

Однак 42 години — це лише задачі, які можна спланувати заздалегідь. У реальності роль Team Lead / DevOps включає значний обсяг робіт, що не піддаються плануванню:

- Багфікси, які виникають у момент інтеграції (наприклад, виправлення критичних помилок RBAC-політик, інвертованої логіки зміни пароля, неправильного біндингу аватарок);

- Архітектурні рішення, які неможливо передбачити до початку розробки (зміна ключового класу Result, що використовується між усіма сервісами — один рефакторинг, який зачіпає весь проект; вибір стратегії кешування TMDB; організація Global Exception Handler);

- Проєктування та деталізація епіків і user stories (~20 годин написання 9 feature-файлів), та їх допрацювання в процесі розробки (з'явився окремий 10-й), коли з'ясовувалось, що початкові специфікації не враховували реальних нюансів реалізації (і це допрацювання неможливо було передбачити на етапі планування);

- code review усіх pull requests обох розробників;

- діагностика та вирішення інфраструктурних проблем (дивна поведінка кешування Docker, повний рефакторинг Docker-файлів).

Реалістична оцінка навантаження TL — **120+ год**, тобто втричі більше формальних 42.

### 5. Аналіз залежностей та блокувань

Ключова проблема — Frontend повністю залежить від Backend. Більшість FE-задач не можуть розпочатися, доки відповідний BE-ендпоінт не написаний та не розгорнутий. Це створює ланцюг блокувань:

Таблиця: Залежності Frontend від Backend

| FE-задача              | Залежить від        | Час очікування (год) |
| ---------------------- | ------------------- |:--------------------:|
| FE-101 Auth Forms      | BE-101 Auth API     | 10                   |
| FE-901 Movie Search UI | BE-901 TMDB API     | 10                   |
| FE-201 Profile Page    | BE-201 Profile CRUD | 6                    |
| FE-801 Reviews UI      | BE-801 Reviews API  | 6                    |
| FE-601 Collections UI  | BE-601 Collections  | 5                    |
| FE-701 Admin Panel     | BE-701 Admin API    | 5                    |

Більше того, на папері FE-801 і BE-603 — паралельні задачі. На практиці — одна людина робить їх послідовно, а інша просто чекає. Поки Пилипчук не закінчив API колекцій, Петренко не може робити UI колекцій.

Лише незначна частина FE-задач може виконуватися незалежно: FE-001 (SvelteKit scaffold), FE-002 (UI Kit setup), FE-401 (Movie card components), FE-602 (Share links), FE-004 (CI для фронтенду) — разом ~40 годин (з junior-коефіцієнтом ~60 год).

### 6. Resource-Leveled прогноз

Грубий resource-leveled прогноз по кожному виконавцю:

- **Пилипчук** працює послідовно: **~138 год** (усі BE-задачі — одна черга)

- **Петренко**: ~60 год незалежних задач (FE-001, FE-002, FE-401, FE-602, FE-004 з коеф. 1,5×), решта **~140 год** заблокована Backend-залежностями

- **Астаф'єв**: формально 42 год, реально **120+ год** (багфікси, архітектура, feature-файли, code review, інфраструктура)

**Реалістичний термін проекту: ~200 людино-годин календарного часу** — обмежений чергою Frontend-розробника-початківця, який додатково заблокований Backend-залежностями. Це в 4 рази більше за CPM-прогноз у 48 годин.

Таблиця: Порівняння CPM і RLS прогнозів

| Параметр                    | CPM      | RLS            |
| --------------------------- |:--------:|:--------------:|
| Прогнозований термін        | 48 год   | ~200 год       |
| Враховує обмеження ресурсів | Ні       | Так            |
| Враховує junior-коефіцієнт  | Ні       | Так (1,5×)     |
| Враховує неплановані роботи | Ні       | Так (TL: 3,0×) |
| Враховує залежності FE↔BE   | Частково | Повністю       |

---

### 7. Обрана стратегія: waterfall зі спринтами

Саме тому в нашому проекті ми використовуємо waterfall зі спринтами — кожен спринт фокусує всю команду на одному епіку, мінімізуючи перемикання контексту та очікування. Пилипчук пише API → Петренко одразу підхоплює UI → Астаф'єв робить code review і тести. І це за ідеального розкладу, не враховуючи хвороби та життєві обставини.

---

### 8. Висновки

У ході виконання лабораторної роботи проведено аналіз термінів завершення проекту «Track List» методом Resource-Leveled Scheduling:

1. **CPM-прогноз** (48 год) передбачає необмежену кількість виконавців і не відображає реальних обмежень команди з 3 людей.

2. **RLS-прогноз** — ~200 людино-годин календарного часу. Різниця у 4 рази пояснюється послідовним виконанням задач одним виконавцем, junior-коефіцієнтом для Frontend-розробника (1,5×) та непланованими роботами TL (коефіцієнт ~3,0×).

3. **Вузьке місце проекту** — Frontend-розробник (Петренко): 134 формальних години × junior-коефіцієнт 1,5 = ~200 год, при цьому більшість задач заблоковані Backend-залежностями.

4. **Стратегія мітигації** — waterfall зі спринтами: послідовна робота над епіками всією командою мінімізує час очікування та перемикання контексту.
