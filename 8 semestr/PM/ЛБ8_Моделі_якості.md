# Лабораторна робота №8

## Моделі якості процесів ЖЦ ПЗ (SPICE) та програмного продукту (SQuaRE)

### 1. Постановка задачі

Метою лабораторної роботи є аналіз моделей якості програмного продукту та процесів життєвого циклу ПЗ. Обрано 4 питання з переліку: SPICE (4), SQuaRE (16), технологічна зрілість процесів (3), критерії оцінки зрілості (7). Огляд побудовано на прикладі проекту «Track List».

### 2. SQuaRE — моделі якості програмного продукту (питання 16)

#### Теоретичний огляд

Серія стандартів ISO/IEC 25000 (SQuaRE — Systems and software Quality Requirements and Evaluation) [1] визначає структурований підхід до оцінки якості ПЗ. Центральний стандарт — **ISO/IEC 25010** [2], який визначає дві моделі:

**Модель якості продукту** — 8 характеристик:

Таблиця: Характеристики якості продукту (ISO/IEC 25010)

| Характеристика         | Опис                                                                                              |
| ---------------------- | ------------------------------------------------------------------------------------------------- |
| Functional Suitability | Ступінь, в якому продукт забезпечує функції, що відповідають заявленим та передбачуваним потребам |
| Performance Efficiency | Продуктивність відносно використаних ресурсів                                                     |
| Compatibility          | Здатність співіснувати та обмінюватись інформацією з іншими системами                             |
| Usability              | Ефективність, продуктивність та задоволеність при використанні                                    |
| Reliability            | Здатність виконувати функції за заданих умов протягом заданого часу                               |
| Security               | Захист інформації та даних від несанкціонованого доступу                                          |
| Maintainability        | Ефективність та продуктивність модифікацій                                                        |
| Portability            | Ефективність перенесення між середовищами                                                         |

**Модель якості у використанні** — оцінює якість з точки зору кінцевого користувача: ефективність, продуктивність, задоволеність, відсутність ризику, покриття контексту використання.

Кожна характеристика декомпозується на підхарактеристики, для яких визначаються метрики вимірювання.

#### Оцінка Track List за SQuaRE

Таблиця: Оцінка Track List за SQuaRE

| Характеристика         | Оцінка для Track List | Обґрунтування                                                                                                                                                                                                                                   |
| ---------------------- | --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Functional Suitability | Висока                | 9 епіків реалізовані відповідно до feature-файлів, BDD-тести підтверджують відповідність                                                                                                                                                        |
| Performance Efficiency | Середня               | Навантажувальне тестування (k6) показало стабільну роботу при 200 одночасних користувачах впродовж 10 хвилин. Пляшкове горлечко — зовнішній TMDB API та кількість контейнерів на одному сервері. Цілеспрямована оптимізація не була пріоритетом |
| Compatibility          | Висока                | REST API + OpenAPI-контракт забезпечують інтеграцію; TMDB API інтегровано                                                                                                                                                                       |
| Usability              | Середня               | UI функціональний, але обмежений досвідом junior-фронтендера                                                                                                                                                                                    |
| Reliability            | Середня               | Docker restart-політики (`unless-stopped`, `on-failure`) забезпечують автоматичний перезапуск, але немає HA (high availability) — один сервер, single point of failure                                                                          |
| Security               | Середня               | JWT + хешування + RBAC, але витік токенів у localStorage був виявлений під час розробки                                                                                                                                                         |
| Maintainability        | Висока                | Тришарова архітектура, DI, Result-патерн, code review — код легко модифікувати                                                                                                                                                                  |
| Portability            | Висока                | Docker-контейнеризація, змінні середовища для конфігурації                                                                                                                                                                                      |

#### Практична цінність SQuaRE

SQuaRE корисна як **чекліст** — структурований спосіб не забути про аспекти якості, які легко випустити з уваги. Без SQuaRE команда Track List природно фокусувалась на Functional Suitability та Maintainability (бо це видно при розробці) і приділяла менше уваги Reliability та Performance (бо це не болить до продакшну).

Водночас SQuaRE не є інструментом прийняття рішень. Оцінка «Reliability: Середня» або «Performance Efficiency: Середня» не говорить, що робити — вона лише фіксує стан. Рішення про інвестицію в HA або оптимізацію залежить від контексту: для дипломного проекту з розгортанням лише на період захисту та стабільною роботою при 200 одночасних користувачах — це не пріоритет.

### 3. Технологічна зрілість процесів ЖЦ (питання 3)

#### Теоретичний огляд

Концепція технологічної зрілості (process maturity / capability) — центральна ідея в менеджменті якості ПЗ. Зрілість процесу визначає, наскільки він формалізований, повторюваний та контрольований.

П'ять рівнів зрілості (SPICE / ISO 15504 [3]):

Таблиця: Рівні зрілості процесів (SPICE)

| Рівень | Назва       | Характеристика                                           |
| ------ | ----------- | -------------------------------------------------------- |
| 0      | Incomplete  | Процес не реалізований або не досягає мети               |
| 1      | Performed   | Процес досягає мети, але не керується формально          |
| 2      | Managed     | Процес планується, відстежується та коригується          |
| 3      | Established | Стандартизований процес, визначений на рівні організації |
| 4      | Predictable | Процес вимірюється кількісно, результати передбачувані   |
| 5      | Optimizing  | Безперервне вдосконалення на основі кількісних даних     |

Логіка моделі: кожен наступний рівень додає ступінь формалізації та контролю. Організація з процесами рівня 5 теоретично здатна передбачувати якість продукту на основі метрик процесів.

#### Оцінка зрілості процесів Track List

Таблиця: Зрілість процесів Track List

| Процес                    | Рівень        | Обґрунтування                                                                              |
| ------------------------- | ------------- | ------------------------------------------------------------------------------------------ |
| Розробка (Implementation) | 1 — Performed | Код пишеться, функціонал реалізується, але без формальних метрик процесу                   |
| Управління конфігурацією  | 2 — Managed   | Git, feature branches, code review перед merge — планований та відстежуваний процес        |
| Тестування                | 1 — Performed | BDD-тести та performance-тести існують, але покриття не вимірюється систематично           |
| Управління проектом       | 1 — Performed | Google-таблиця, GitHub Issues, Telegram — проект керується, але не за стандартним процесом |
| Code Review               | 2 — Managed   | 100% PR проходять ревью, є неформальні критерії якості, результати відстежуються           |

Середня зрілість процесів Track List — **між 1 та 2**. Процеси існують і досягають мети, частина з них планується та відстежується. Жоден не стандартизований на рівні організації (рівень 3), бо «організація» — це три студенти.

#### Зрілість vs результат

Track List має тришарову архітектуру, Result-патерн, BDD-тести, code review — і все це при зрілості процесів 1–2. Модель зрілості описує **як організовано процес**, а не **який результат він дає**. Це принципове обмеження, до якого повернемось у секції 5.

### 4. SPICE — оцінювання процесів ЖЦ ПЗ (питання 4)

#### Теоретичний огляд

**SPICE (Software Process Improvement and Capability Determination)** — серія стандартів ISO/IEC 15504:2004 [3], що визначає фреймворк для оцінки зрілості процесів ЖЦ ПЗ. SPICE є основою «повільного» контуру регулювання в стратегії TQM: якщо PDCA коригує якість в межах проекту, SPICE підвищує зрілість процесів на рівні організації.

Структура SPICE:

- **Dimension 1: Process Dimension** — які процеси оцінюються (з ISO 12207 [4]).
- **Dimension 2: Capability Dimension** — на якому рівні зрілості знаходиться кожен процес (0–5).

Для кожного рівня визначено **атрибути процесу (Process Attributes, PA)**:

Таблиця: Атрибути процесу за рівнями

| Рівень | Атрибути                                                      |
| ------ | ------------------------------------------------------------- |
| 1      | PA 1.1 Process Performance                                    |
| 2      | PA 2.1 Performance Management, PA 2.2 Work Product Management |
| 3      | PA 3.1 Process Definition, PA 3.2 Process Deployment          |
| 4      | PA 4.1 Process Measurement, PA 4.2 Process Control            |
| 5      | PA 5.1 Process Innovation, PA 5.2 Process Optimization        |

Кожен атрибут оцінюється за шкалою: N (Not achieved, 0–15%), P (Partially, 15–50%), L (Largely, 50–85%), F (Fully, 85–100%).

Процес досягає рівня N, якщо всі атрибути рівнів 1..N оцінені як L або F, а атрибути рівня N — як F.

#### Застосування SPICE до Track List

Формальна SPICE-оцінка вимагає сертифікованого асесора та документованої процедури. Неформальна самооцінка:

Таблиця: Самооцінка за SPICE

| Атрибут                        | Оцінка           | Обґрунтування                                                                 |
| ------------------------------ | ---------------- | ----------------------------------------------------------------------------- |
| PA 1.1 Process Performance     | L (Largely)      | Процеси досягають мети: код пишеться, тести проходять, PR мержаться           |
| PA 2.1 Performance Management  | P (Partially)    | Є планування (Google-таблиця, спринти), але без формальних метрик відстеження |
| PA 2.2 Work Product Management | L (Largely)      | Git забезпечує версіонування, code review — контроль якості артефактів        |
| PA 3.1 Process Definition      | N (Not achieved) | Процеси не визначені як стандарт організації — вони існують «в головах»       |
| PA 3.2+                        | N                | Не застосовується                                                             |

Результат: процеси Track List знаходяться на **рівні 1** за SPICE, з елементами рівня 2 (управління конфігурацією).

#### Обмеження SPICE для малих команд

SPICE розроблений для організацій, де процеси можуть «загубитись» через масштаб. В команді з 3 осіб процеси не можуть бути «не визначені» в практичному сенсі — кожен знає, як працює команда, бо бачить весь процес. Формальне визначення процесу (PA 3.1) для Track List означало б написання документа, який описує те, що всі і так знають. Це корисна вправа для навчання, але не для підвищення якості продукту.

### 5. Критерії оцінки технологічної зрілості (питання 7)

#### Теоретичний огляд

Оцінка зрілості за SPICE базується на **профілі рейтингів атрибутів (ПРА)** процесу. ПРА — це набір оцінок (N/P/L/F) для всіх атрибутів процесу, візуалізований як стовпчаста або радіальна діаграма.

Ключовий критерій зрілості — **розмір «плато»** профілю: ширина горизонтальної частини ПРА, де всі атрибути мають однаково високі оцінки. Широке плато означає рівномірну зрілість, вузьке — нерівномірну (одні аспекти розвинуті, інші відстають).

Для формалізації оцінки пропонується використання:

- **Радіально-метричних діаграм (РМД)** — візуалізація багатокомпонентних показників якості у вигляді полігону на радіальних осях.
- **Площі та зміщення центру ваги РМД** — інтегральні метрики, що згортають багатовимірну оцінку в одне число.
- **Нечіткого регулятора** — математичний апарат теорії нечітких множин для управління процесом підвищення зрілості в контурі SPICE системи менеджменту якості.

Ідея: рейтинги атрибутів процесу є лінгвістичними змінними (N/P/L/F), які подаються на вхід нечіткого регулятора. Регулятор визначає, які базові практики менеджменту застосувати для підвищення зрілості. Параметри регулювання: обов'язковість виконання, наявність ресурсів, верифікація виконання, поліпшення.

#### Критичний аналіз

ПРА як діагностика — працює. Проблеми починаються, коли ця концепція обростає математичним апаратом: нечіткими регуляторами, радіально-метричними діаграмами, згорткою ієрархій та оптимізацією площі полігону.

**Проблема адекватності моделі.** Нечіткий регулятор передбачає, що зрілість процесів — це контрольована система зі зворотним зв'язком, аналогічна технічній системі (термостату, автопілоту). Але процеси ЖЦ ПЗ виконуються людьми, а не механізмами. Людські процеси мають нелінійну динаміку: впровадження code review може миттєво підвищити якість коду, а впровадження daily standup — не змінити нічого або навіть погіршити продуктивність через переривання потоку роботи. Нечіткий регулятор не здатний моделювати такі стрибкоподібні та контекстно-залежні зміни.

**Проблема вимірюваності.** Рейтинги N/P/L/F — суб'єктивні експертні оцінки з похибкою ±30%, які два асесори можуть виставити по-різному. Площа РМД, обчислена з точністю до десятих на основі таких оцінок — ілюзія точності.

**Проблема вартості.** Формальна SPICE-оцінка (як і CMMI-оцінка) — це дорога процедура. Для великих організацій це інвестиція в передбачуваність. Для малих команд — непропорційна витрата.

#### Індустріальний кейс: CMMI Level 5 як формальність

Найвідоміший приклад розриву між формальною зрілістю процесів та реальною якістю продукту — масова CMMI Level 5 сертифікація індійських IT-аутсорсерів.

**CMMI (Capability Maturity Model Integration)** [5] — модель зрілості, концептуально близька до SPICE, з 5 рівнями від Initial до Optimizing. Level 5 — найвищий, означає безперервне вдосконалення процесів на основі кількісних даних.

Компанії Infosys, Wipro, TCS, HCL та інші великі індійські аутсорсери отримали CMMI Level 5 для значної частини своїх підрозділів. Формально — це найвищий рівень зрілості процесів у індустрії. На практиці якість коду, що постачається цими компаніями, стала стійким об'єктом критики в IT-спільноті: технічний борг, необхідність повного переписування після передачі проекту in-house, невідповідність специфікаціям.

Причини розриву:

- **Сертифікація оцінює процеси, а не продукт.** CMMI Level 5 підтверджує, що організація має документовані, вимірювані та безперервно вдосконалювані процеси. Він не підтверджує, що код, написаний у рамках цих процесів, працює правильно.
- **Goodhart's Law** [6]**.** «Коли міра стає ціллю, вона перестає бути хорошою мірою.» Коли CMMI-сертифікація стала конкурентною перевагою для отримання контрактів, компанії оптимізували свої процеси під проходження оцінки, а не під виробництво якісного ПЗ.
- **Масштаб vs контроль.** В організаціях з десятками тисяч розробників формальні процеси існують на папері, але їх дотримання на рівні конкретної команди — питання локальної культури, а не сертифікату.

Аналогічна ситуація з **ISO 9001** [7] у software: сертифікація підтверджує наявність задокументованої системи менеджменту якості, але не якість самого ПЗ. Компанія може мати ідеальну документацію процесів та систематично випускати баговий код.

### 6. Висновки

SQuaRE та SPICE — два боки одного підходу: якість продукту (що робимо) та якість процесів (як робимо). Обидві моделі корисні як діагностичний інструмент, але математична надбудова (нечіткі регулятори, РМД, згортка ієрархій) додає складність без пропорційного зростання точності. CMMI-кейс та ISO 9001 підтверджують: сертифікат на стіні не замінює компетентність команди.
