### 2. Класифікація методів верифікації ПЗ. Місце і роль статичного аналізу (питання 15)

#### Спектр верифікації

Верифікація ПЗ — підтвердження, що продукт відповідає специфікації [1]. Методи верифікації утворюють спектр від неформальних до математично строгих:

Таблиця: Спектр методів верифікації

| Метод                                   | Формальність | Що перевіряє                                             | Коли виявляє дефект |
| --------------------------------------- | ------------ | -------------------------------------------------------- | ------------------- |
| Code review                             | Низька       | Логіка, стиль, архітектурні рішення                      | До компіляції       |
| Статичний аналіз                        | Середня      | Типові помилки, порушення правил, потенційні вразливості | До компіляції       |
| Модульне тестування                     | Середня      | Коректність окремих одиниць коду                         | Після компіляції    |
| Інтеграційне тестування                 | Середня      | Взаємодія компонентів                                    | Після компіляції    |
| BDD/acceptance тестування               | Середня      | Відповідність бізнес-вимогам                             | Після компіляції    |
| Навантажувальне тестування              | Середня      | Поведінка під навантаженням                              | Після розгортання   |
| Model checking                          | Висока       | Відповідність формальній моделі (темпоральна логіка)     | До реалізації       |
| Формальна верифікація (theorem proving) | Дуже висока  | Математичний доказ коректності                           | До реалізації       |

Кожен рівень формальності має свою ціну. Code review — дешевий, але залежить від уважності рев'юера. Формальна верифікація — дорога, але дає математичні гарантії. Більшість комерційних проектів працює в середній зоні: статичний аналіз + тестування + code review.

#### Статичний аналіз у спектрі

Статичний аналіз займає унікальну позицію: він автоматичний (на відміну від code review), працює без запуску програми (на відміну від тестування) і масштабується на будь-який розмір кодової бази. Це єдиний метод верифікації, який може перевірити 100% коду за прийнятний час — тести покривають лише ті шляхи виконання, для яких написані.

Обмеження: статичний аналіз працює з текстом програми, а не з її поведінкою. Він може знайти null dereference, але не може знайти помилку бізнес-логіки «рецензія зберігається до створення акаунта». Для цього потрібні BDD-тести або формальна специфікація.

#### Верифікація в Track List

Track List використовує три рівні верифікації:

1. **Статичний аналіз** — SonarAnalyzer.CSharp, EF Core Analyzers, ESLint (детально в секції 3).
2. **BDD-тестування** — Reqnroll (C#) + Cucumber (frontend) з Gherkin-сценаріями для всіх 9 епіків.
3. **Code review** — 100% PR проходять ревью перед merge.

Навантажувальне тестування (k6) додає четвертий рівень — верифікацію нефункціональних вимог. Формальна верифікація та model checking не застосовуються — ціна невідповідна рівню критичності застосунку.

### 3. Статичний аналіз: цілі, можливості, обмеження (питання 16)

#### Що робить статичний аналізатор

Статичний аналізатор — інструмент, що аналізує вихідний код без його виконання. На вхід отримує AST (Abstract Syntax Tree) або семантичну модель програми, на виході видає діагностики — повідомлення про потенційні проблеми.

Категорії діагностик:

Таблиця: Категорії діагностик статичного аналізу

| Категорія       | Приклад                                                | Цінність                             |
| --------------- | ------------------------------------------------------ | ------------------------------------ |
| Помилки         | Null dereference, unreachable code, use after dispose  | Висока — запобігає runtime-помилкам  |
| Стиль           | Naming conventions, зайві using, порядок модифікаторів | Середня — забезпечує консистентність |
| Performance     | Зайві алокації, boxing, неефективні LINQ-запити        | Середня — залежить від контексту     |
| Security        | SQL injection, XSS, hardcoded credentials              | Висока — запобігає вразливостям      |
| Maintainability | Cyclomatic complexity, method length, coupling         | Низька–середня — суб'єктивна         |

#### Можливості та обмеження

**Що статичний аналіз робить добре:**

- Знаходить типові помилки швидше за людину — жоден рев'юер не тримає в голові сотні патернів, які перевіряє аналізатор автоматично.
- Завжди доступний. Завжди працює.
- Виявляє проблеми до запуску. Помилка, знайдена при написанні коду, в десятки разів дешевша за помилку в продакшні.

**Чого статичний аналіз не може:**

- Розуміти бізнес-логіку. Аналізатор бачить, що `Result.Value` читається — він не знає, чи перевірив розробник `IsSuccess` десятьма рядками вище в іншому методі.
- Гарантувати відсутність помилок. Теорема Райса [2]: будь-яка нетривіальна семантична властивість програми є нерозв'язною. Статичний аналіз працює з наближеннями — і тому генерує false positives (помилкові спрацювання) та false negatives (пропущені помилки).
- Замінити тестування. Аналізатор перевіряє структуру коду, тест перевіряє поведінку програми — це протилежні методи.

#### False positives: реальна проблема

False positives — найбільша практична проблема статичного аналізу. Коли аналізатор генерує занадто багато хибних спрацювань, розробники починають ігнорувати всі попередження.

Баланс між повнотою (знайти всі помилки) та точністю (не генерувати хибних) — фундаментальний компроміс. Строгі аналізатори (як ті, що використовуються в авіації за DO-178C [3]) налаштовані на повноту: краще 100 false positives, ніж один пропущений дефект. Комерційні аналізатори (SonarQube, ESLint) налаштовані на точність: краще пропустити дефект, ніж дратувати розробника.

#### Статичний аналіз в Track List

**Backend (C#/.NET):**

- **SonarAnalyzer.CSharp** [4] (v10.23.0) — Roslyn-аналізатор від SonarSource. ~500 правил для C#: null dereference, dead code, security hotspots, code smells. Інтегрований як NuGet-пакет, працює при кожній компіляції.
- **Microsoft.EntityFrameworkCore.Analyzers** — специфічні правила для EF Core: попередження про N+1 запити, неправильне використання Include(), проблеми з міграціями.
- **JetBrains.Annotations** — атрибути `[NotNull]`, `[CanBeNull]`, `[UsedImplicitly]` тощо. Не аналізатор сам по собі, а метадані для ReSharper/Rider, які підвищують точність їхнього аналізу.
- **Qodana** [5] — CI/CD інструмент від JetBrains для статичного аналізу. Конфігурація `qodana.yaml` з профілем QDNET. Агрегує результати Roslyn-аналізаторів та власних інспекцій ReSharper.

**Frontend (SvelteKit/TypeScript):**

- **ESLint** [6] — лінтер з плагінами для TypeScript (`@typescript-eslint`) та Svelte (`eslint-plugin-svelte`). Конфігурація в `.eslintrc.cjs`. Інтеграція з Prettier через `eslint-config-prettier` для уникнення конфліктів правил форматування.
- **Prettier** — форматер коду. Конфігурація в `.prettierrc`: tabs, 100 символів, Svelte-плагін. Не є аналізатором — лише забезпечує єдиний стиль.
- **TypeScript compiler** (`tsc --noEmit`) — строга типізація як форма статичного аналізу. `strict: true` вмикає null checks, implicit any detection та інші перевірки.

**Що це дає на практиці:**

Три рівні аналізу працюють каскадно: Roslyn підсвічує помилку в IDE миттєво при написанні, ESLint автоматично виправляє стилістику при збереженні файлу, Qodana перевіряє всю кодову базу в CI. Стилістичні зауваження, забуті null-перевірки, неефективні запити — все ловиться до створення PR.

### 4. Інструментальні засоби і середовища (питання 22)

Track List — не академічний приклад, а реальний проект з конкретним набором інструментів. Кожен інструмент обраний з практичних причин, а не для заповнення чекліста стандарту.

**Середовище розробки:**

| JetBrains Rider   | IDE для C#/.NET backend    | Астаф'єв (TL), Пилипчук |

#### Обмеження інструментального підходу

Інструменти вирішують конкретні проблеми, але створюють нові:

**Конфігураційний борг.** Кожен інструмент має свій конфіг: `.eslintrc.cjs`, `.prettierrc`, `.editorconfig`, `qodana.yaml`, `tsconfig.json`. Конфігурації можуть конфліктувати — ESLint та Prettier мають перетин правил форматування, для чого потрібен `eslint-config-prettier`. Підтримка консистентності між конфігами — окрема задача.

**Залежність від екосистеми.** Track List використовує інструменти двох екосистем: JetBrains (Rider, ReSharper, Qodana, DataGrip) та VS Code (ESLint, Prettier, Svelte Language Server). Правила в `.editorconfig` розуміються Rider, але не VS Code без додаткових розширень. Це створює ситуацію, коли backend- та frontend-розробники фактично працюють у різних інструментальних середовищах з різними правилами.

#### Інструменти vs оточення

Інструмент працює лише в оточенні, що підтримує його використання. ESLint з 200 правилами не допоможе, якщо розробник додає `// eslint-disable-next-line` перед кожним попередженням. SonarAnalyzer не допоможе, якщо severity всіх діагностик знижено до «suggestion». Qodana не допоможе, якщо ніхто не дивиться на звіт. Будь-яка дисципліна розсипеться в оточенні, де результат її дотримання не видно або не цінується.

В Track List оточення побудоване так, що suppress без пояснення — видимий і помітний. Малий розмір команди та спільний контекст роблять обхід правил складнішим за їх дотримання.

### 5. Нормативна база інженерії критичного ПЗ (питання 25)

#### Ландшафт стандартів

Інженерія критичного ПЗ регулюється ієрархією стандартів від загальних до галузевих:

Таблиця: Ландшафт стандартів інженерії критичного ПЗ

| Рівень     | Стандарт                    | Що регулює            |
| ---------- | --------------------------- | --------------------- |
| Загальний  | ISO/IEC/IEEE 12207:2008 [7] | Процеси ЖЦ ПЗ         |
| Загальний  | ISO/IEC 25010 (SQuaRE) [8]  | Якість продукту       |
| Загальний  | ISO/IEC 15504 (SPICE) [9]   | Зрілість процесів     |
| Авіація    | DO-178C [3]                 | Бортове ПЗ            |
| Медицина   | IEC 62304 [10]              | ПЗ медичних пристроїв |
| Автомобілі | ISO 26262 [11]              | Функціональна безпека |
| Атомна     | IEC 60880 [12]              | ПЗ АСУ ТП АЕС         |
| Космос     | ECSS-E-ST-40C [13]          | ПЗ космічних систем   |
| Залізниця  | EN 50128 [14]               | Залізничні системи    |

Логіка стандартизації зрозуміла: чим вища ціна дефекту, тим жорсткіші вимоги до процесу розробки. DO-178C [3] вимагає 100% покриття структурних критеріїв для ПЗ рівня A (катастрофічні наслідки відмови). IEC 62304 [10] класифікує ПЗ за рівнем безпеки (A/B/C) і пропорційно підвищує вимоги.

#### Проблема стандартизації: коли уніфікація створює хаос

Стандартизація має фундаментальну проблему: вона працює лише коли всі учасники однозначно розуміють стандарт. Коли стандарт допускає неоднозначне тлумачення або маркування, уніфікація перетворюється на джерело плутанини.

**USB Type-C: уніфікація роз'єму, хаос протоколів.**

USB Type-C мав стати єдиним роз'ємом для всього: заряджання, передача даних, відео. Фізичний роз'єм дійсно уніфікований — один конектор замість десятка попередників. Але за однаковим роз'ємом ховаються принципово різні можливості:

- USB 2.0 (480 Мбіт/с) і USB4 (40 Гбіт/с)
- Кабель може підтримувати або не підтримувати Power Delivery (від 15 Вт до 240 Вт).
- Thunderbolt 3 та 4 використовує той самий Type-C, але вимагає активний кабель.
- DisplayPort Alt Mode працює через Type-C, але не кожен порт його підтримує.
- Маркування: логотип «SS» (SuperSpeed), число «10» або «20» біля порту, символ блискавки для Thunderbolt — або повна відсутність маркування.

Результат: звичайний користувач купує кабель USB-C, підключає зовнішній монітор — не працює. Купує іншій — працює, але лише 1080p. Третій кабель дає 4K, бо підтримує DisplayPort Alt Mode. Всі три кабелі мають однаковий роз'єм. Стандарт уніфікував фізичну форму, але не забезпечив однозначного маркування можливостей — і створив ситуацію, гіршу за попередню, коли різні роз'єми хоча б візуально відрізнялись.

Це пряма аналогія з ISO 12207: стандарт описує процеси ЖЦ, але конкретна організація може «реалізувати» їх з принципово різним рівнем якості — і формально відповідати тому ж стандарту.

**X11: стандарт, що заблокував розвиток на 30 років.**

X Window System (X11) [15] — стандарт графічної підсистеми Unix/Linux, створений у 1987 році. Архітектура X11 базується на клієнт-серверній моделі з мережевою прозорістю: застосунок (клієнт) надсилає команди X-серверу через протокол, сервер відображає графіку. Це було інноваційно для мережевих робочих станцій 1980-х.

Проблема: ця архітектура від самого початку не була призначена для спілкування ядра та монітору. Її базова ідея була ближче до функціоналу застосунку AnyDesk. Але її стандартизованість та вкоріненість заблокували перехід на нову модель:

- **Безпека.** X11 не має ізоляції між клієнтами. Будь-який застосунок може перехопити введення з клавіатури будь-якого іншого застосунку. Кейлогер на X11 — тривіальна задача. Протокол проектувався для довіреного середовища університетських мереж, не для сучасного десктопу з браузером та месенджерами.
- **Продуктивність.** Мережевий протокол між застосунком та дисплеєм додає latency навіть при локальному з'єднанні. Композитинг (прозорість, анімації, HiDPI) реалізований як хаки поверх протоколу 1987 року.
- **Рендер різних моніторів, як одне зображення.** Протокол жахливо працює з кількома дисплеями. Х11 по дизайну не може мати кілька різних масштабувань на одному дисплей сервері. Це значить, що якщо у вас є 1080p та 4К монітори, то у вас буде або замале (125%) масштабування, або завелике (200%) на одному із них. 
- **Фрагментація.** Відсутність стандартного способу управління вікнами та введенням (input) породила десятки window manager-ів та display manager-ів з несумісними розширеннями. Кожен Desktop Environment (GNOME, KDE, XFCE) фактично реалізував свій шар абстракції над X11.

Wayland [16] (2008→) замінює X11 прямою комунікацією між compositor та клієнтом, з ізоляцією та нативним HiDPI. Але перехід зайняв більше 15 років — через масивну кількість ПЗ, залежного від X11 API. Стандарт, що мав забезпечити стабільність, забезпечив стагнацію.

Паралель із ПЗ: ISO/IEC 12207:2008 описує процеси ЖЦ у термінах, актуальних для waterfall-ери. DevOps, CI/CD, Infrastructure as Code — практики, що з'явились після стандарту. Команда, що впровадила CD (Continuous Deployment), формально «не відповідає» стандартним процесам верифікації та валідації — хоча на практиці випускає якісніший код, ніж команда з формально правильним process gate. При тому, що концепція waterfall майже ідеально підходить для малих команд.

#### Стандартизація та Track List

Track List — не критичний застосунок. Дефект в рецензії на фільм не призведе до жертв. Формально проект не підпадає під жоден галузевий стандарт.

Але елементи стандартизованих практик використовуються прагматично:

- **Верифікація** (ISO 12207 [7], 7.2.4) — реалізована через BDD-тести та статичний аналіз, без формального плану верифікації.
- **Управління конфігурацією** (ISO 12207 [7], 7.2.2) — Git з feature branches та code review, без Configuration Management Plan.
- **Забезпечення якості** (ISO 12207 [7], 7.2.3) — code review кожного PR, без Quality Assurance Plan.

Різниця між Track List та критичним ПЗ — не у відсутності практик, а у відсутності документального підтвердження їх виконання. В авіації DO-178C вимагає не лише виконати верифікацію, а й задокументувати кожен крок: що перевірялось, ким, коли, з яким результатом. Для малих, динамічних проєктів ця внутрішня докуметнація буде з’їдати людино–години не даючи приріст до швидкості розробки або якості фінального продукту.

### 6. Висновки

Статичний аналіз — найбільш практичний метод верифікації для типового веб-застосунку: автоматичний, масштабований, інтегрований в IDE та CI. В спектрі верифікації він займає зону між code review та тестуванням — доповнює обидва, не замінюючи жодного.

Стандартизація працює, коли визначає чіткі вимоги пропорційно ціні дефекту. Стандарт, що уніфікує форму без змісту — як USB Type-C без маркування або X11 з архітектурою 1987 року — стає перешкодою. Track List запозичує практики зі стандартів без бюрократичної обв'язки — і це свідомий вибір, а не недолік.
