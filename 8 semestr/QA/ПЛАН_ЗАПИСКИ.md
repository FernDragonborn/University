# План: Записка до курсачу ТВПЗ(КП) — Performance Testing

## Контекст

Моя частина курсового проєкту — **тестування продуктивності** (performance testing) застосунку Track List. Це окремий розділ, не прив'язаний до ПР1-8. Додатково — обґрунтування вибору інструментів.

Результати вже є: `D:\data\code\group-diploma-track-list\Source\Tests\Performance\`

---

## Структура документа

### 1. Обґрунтування вибору інструментів тестування продуктивності

- Чому k6 (не JMeter, Gatling, Artillery):
  - JS-based → один стек з фронтендом Track List
  - CLI-first → легко інтегрувати в CI/CD
  - Написаний на Go → низьке споживання ресурсів самим тест-раннером
  - Вбудовані thresholds → автоматичний pass/fail
  - HTML-звітність через k6-reporter
- Чому dotnet-counters для серверних метрик:
  - Нативний для ASP.NET Core
  - CPU, RAM, GC, threadpool, requests/sec без зовнішніх агентів
- Docker Compose для ізольованого середовища
- Caddy як reverse proxy (prod-like)

### 2. Планування тестування продуктивності

- **Мета**: оцінити часові характеристики, використання ресурсів, пропускну здатність (ISO/IEC 25010 — Performance Efficiency: Time behaviour, Resource utilization, Capacity)
- **Об'єкт тестування**: REST API Track List (ASP.NET Core + PostgreSQL + Redis)
- **Тестове середовище**: Docker Compose (API, DB, Redis, Caddy), localhost
- **Seed data**: 24 users, 20 media, 60 reviews, 120 comments, 100 likes, 10 collections, 40 tracking statuses
- **Ендпоінти під тестом**:
  - Anonymous: feed/global, media/search, media/{id}, collections/public
  - Authenticated: feed/personal, reviews CRUD, likes, comments, tracking, collections
  - Admin: stats, CSV export
  - Moderator: pending translations
- **Три сценарії**:

| Сценарій | VUs                              | Тривалість | Мета                                                              |
| -------- | -------------------------------- | ---------- | ----------------------------------------------------------------- |
| Smoke    | 2                                | 1 хв       | Базова перевірка — чи працює все під мінімальним навантаженням    |
| Load     | 50 (30 anon + 15 auth + 5 admin) | 5 хв       | Типове навантаження — нормальний трафік                           |
| Stress   | 10→50→100→200→0 (ramping)        | 10 хв      | Пошук точки деградації — де починають рости латентність і помилки |

- **Thresholds (критерії прийняття)**:

| Сценарій | p(95)   | p(99) | Max Error Rate |
| -------- | ------- | ----- | -------------- |
| Smoke    | < 500ms | < 1s  | < 1%           |
| Load     | < 1s    | < 2s  | < 5%           |
| Stress   | < 2s    | < 5s  | < 10%          |

### 3. Виконання тестування продуктивності

- **Тестове середовище**:
  - ОС: Windows 10 (WSL2/Docker Desktop)
  - Ресурси WSL: memory=8GB; processors=6; swap=4GB
  - Інструмент: k6 v0.x + dotnet-counters
  - Стек: ASP.NET Core, PostgreSQL, Redis, Caddy
  - Дата виконання: 27.04.2026
- **Результати Smoke** (2 VUs, 1 min):
  - 733 HTTP requests, 81 iterations
  - p(95) = 148ms (threshold <500ms — **PASS**)
  - Error rate: 0% (threshold <1% — **PASS**)
  - Усі checks pass: 733/733
- **Результати Load** (50 VUs, 5 min):
  - 16168 HTTP requests, 5388 iterations, ~57 req/s
  - p(95) = 157ms (threshold <1000ms — **PASS**)
  - Error rate: 0% (threshold <5% — **PASS**)
  - Усі checks pass: 16168/16168
- **Результати Stress** (10→200 VUs, 10 min):
  - 84975 HTTP requests, 35411 iterations, ~151 req/s
  - p(95) = 105ms (threshold <2000ms — **PASS**)
  - Error rate: 0% (threshold <10% — **PASS**)
  - Усі checks pass: 84975/84975
- **Зведена таблиця**:

- Рисунок: HTML-звіт smoke-тестування (загальна статистика та графік латентності)
- Рисунок: HTML-звіт load-тестування (розподіл запитів по сценаріях)
- Рисунок: HTML-звіт stress-тестування (графік VUs та латентності при зростанні навантаження)
- Рисунок: Серверні метрики dotnet-counters під час load-тесту (CPU, RAM, GC)
- Рисунок: Зведена таблиця результатів (скріншот терміналу k6)
- **Bottleneck**: при даному навантаженні вузьким місцем є rate limit API TMDB (зовнішній сервіс), а не сам Track List API

### 4. Висновки тестування продуктивності

- Усі thresholds пройдені для всіх трьох сценаріїв
- Навіть при 200 VUs p(95) = 105ms — значний запас
- Error rate = 0% на всіх рівнях навантаження
- Track List API здатний обробляти ~151 req/s без деградації
- Bottleneck — rate limit TMDB API, не внутрішня інфраструктура
- Рекомендації: CDN для статики, кешування відповідей TMDB, connection pooling tuning
