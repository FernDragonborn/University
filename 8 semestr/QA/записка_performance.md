# Тестування продуктивності програмного забезпечення Track List

## 1 Обґрунтування вибору інструментів тестування продуктивності

Для проведення тестування продуктивності веб-застосунку Track List було обрано інструмент **k6** [2] — сучасну платформу для навантажувального тестування з відкритим вихідним кодом, розроблену компанією Grafana Labs. Вибір k6 обумовлений такими перевагами порівняно з альтернативними рішеннями:

Таблиця: Порівняння інструментів тестування продуктивності

| Критерій                   | k6                        | JMeter            | Gatling            | Artillery         |
| -------------------------- | ------------------------- | ----------------- | ------------------ | ----------------- |
| Мова сценаріїв             | JavaScript (ES6+)         | XML / GUI         | Scala / Java       | YAML / JS         |
| Споживання ресурсів        | Низьке (Go runtime)       | Високе (JVM)      | Середнє (JVM)      | Середнє (Node.js) |
| Інтеграція з CI/CD         | CLI-first, вбудована      | Потребує плагінів | Через sbt/Maven    | CLI               |
| Пороги якості (thresholds) | Вбудовані, декларативні   | Потребує плагінів | Через assertions   | Обмежені          |
| HTML-звітність             | k6-reporter               | Вбудована         | Вбудована          | Обмежена          |
| Підтримка сценаріїв        | Executors, stages, groups | Thread groups     | Injection profiles | Phases            |

Ключові переваги k6 для проєкту Track List:

1. **Єдиний технологічний стек** — сценарії тестів написані мовою JavaScript, що збігається з мовою фронтенд-частини застосунку (TS/Sveltekit).

2. **Мінімальне споживання ресурсів** — ядро k6 написане мовою Go, що дозволяє генерувати значне навантаження (сотні віртуальних користувачів) з одного хосту без впливу самого тест-раннера на результати вимірювань.

3. **Декларативні пороги якості (thresholds)** — дозволяють визначити критерії прийняття (наприклад, `p(95) < 300ms`, `error rate < 1%`) безпосередньо у конфігурації тесту, забезпечуючи автоматичне визначення результату pass/fail [4].

4. **Гнучка модель навантаження** — підтримка різних типів executors (`constant-vus`, `ramping-vus`, `per-vu-iterations`) дозволяє моделювати як стабільне, так і зростаюче навантаження в межах одного інструменту.

Для збору серверних метрик обрано **dotnet-counters** [3] — вбудований діагностичний інструмент платформи .NET, що дозволяє у реальному часі відстежувати ключові показники ASP.NET Core застосунку без встановлення додаткових агентів:

- використання CPU процесом API;
- обсяг оперативної пам'яті (working set);
- розмір керованої купи (GC heap) та частота збирання сміття по поколіннях;
- кількість активних потоків та черга thread pool;
- кількість запитів на секунду та кількість помилок (5xx).

Тестове середовище розгорнуто за допомогою **Docker Compose** [7], що забезпечує ізольованість та відтворюваність результатів. Як reverse proxy використано **Caddy**, що наближує архітектуру тестового середовища до продуктивної конфігурації.

## 2 Планування тестування продуктивності

### 2.1 Мета тестування

Метою тестування продуктивності є оцінювання характеристик якості застосунку Track List за стандартом ISO/IEC 25010 [1] у категорії **Performance Efficiency (Продуктивність)**:

- **Time behaviour** (часові характеристики) — час відгуку API на запити користувачів;
- **Resource utilization** (використання ресурсів) — споживання CPU, оперативної пам'яті та з'єднань з базою даних;
- **Capacity** (пропускна здатність) — максимальна кількість одночасних користувачів та запитів, які застосунок обробляє без деградації якості.

### 2.2 Об'єкт тестування

Об'єктом тестування є REST API застосунку Track List, побудований на платформі ASP.NET Core [6] з використанням СУБД PostgreSQL [8]. Застосунок надає API для управління мультимедійним контентом, рецензіями, колекціями, соціальними взаємодіями та адмініструванням.

### 2.5 Сценарії тестування

Розроблено чотири сценарії тестування [4, 10], що послідовно збільшують навантаження:

Таблиця: Сценарії тестування продуктивності

| Сценарій     | Віртуальні користувачі (VUs)     | Тривалість | Мета                                                                                                                                |
| ------------ | -------------------------------- | ---------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Smoke        | 2                                | 1 хв       | Базова перевірка працездатності всіх ендпоінтів під мінімальним навантаженням                                                       |
| Load         | 50 (30 anon + 15 auth + 5 admin) | 5 хв       | Симуляція типового трафіку з розподілом за ролями: 60% анонімних, 30% автентифікованих, 10% адміністративних запитів                |
| Stress       | 10→50→100→200→0 (ramping)        | 10 хв      | Пошук точки деградації — поступове збільшення навантаження для визначення межі, за якою латентність зростає або з'являються помилки |
| Ultra-Stress | 50→200→500→1000→0 (ramping)      | 15 хв      | Визначення абсолютної межі системи — екстремальне навантаження до 1000 одночасних користувачів                                      |

Розподіл трафіку у stress-сценарії: 70% анонімних запитів, 20% автентифікованих, 10% адміністративних (зважена випадкова вибірка). У ultra-stress-сценарії розподіл зміщено на користь операцій читання: 80% анонімних, 15% автентифікованих, 5% адміністративних, що відповідає реалістичному патерну трафіку при екстремальному навантаженні.

**Smoke-тестування** — «димовий тест», перевіряє базову працездатність під мінімальним навантаженням (2 VUs).

**Load-тестування** — моделює робоче навантаження з розподілом за трьома executor: анонімні, автентифіковані та адміністративні запити.

**Stress-тестування** — використовує `ramping-vus` для пошуку точки деградації при зростанні від 10 до 200 VUs.

**Ultra-stress-тестування** — визначає абсолютну межу системи: 50→200→500→1000→500→0 VUs. Пороги послаблені — мета не відповідність критеріям, а характер деградації.

### 2.6 Критерії прийняття (thresholds)

Для кожного сценарію визначено пороги якості [5], при перевищенні яких тест вважається непройденим:

Таблиця: Критерії прийняття тестування продуктивності

| Сценарій     | p(50) latency | p(95) latency | p(99) latency | Макс. частка помилок | p(95) admin export | p(95) write |
| ------------ | ------------- | ------------- | ------------- | -------------------- | ------------------ | ----------- |
| Smoke        | < 100 мс      | < 300 мс      | < 500 мс      | < 1%                 | < 1500 мс          | —           |
| Load         | < 100 мс      | < 300 мс      | < 1000 мс     | < 5%                 | < 3000 мс          | < 1500 мс   |
| Stress       | < 100 мс      | < 300 мс      | < 2000 мс     | < 10%                | < 5000 мс          | < 3000 мс   |
| Ultra-Stress | < 300 мс      | < 1000 мс     | < 5000 мс     | < 20%                | —                  | < 5000 мс   |

Додано поріг p(50) — медіанна латентність — для точнішого контролю типового часу відгуку. Пороги ultra-stress-сценарію суттєво послаблені, оскільки цей рівень навантаження (1000 VUs) значно перевищує очікуване робоче навантаження. Окремі пороги для admin export та write-операцій обумовлені їх підвищеною ресурсоємністю (формування CSV-звіту, запис у базу даних з валідацією).

### 3.8 Аналіз динаміки серверних метрик

Збір метрик dotnet-counters проводився кожну секунду протягом усього тесту. Аналіз динаміки дозволяє виявити фази поведінки системи та моменти переходу між ними.

**Smoke-тестування (2 VUs, 1 хв).** Метрики залишаються стабільними протягом усього тесту. CPU коливається в межах 11–21%, Working Set стабільний (~246 МБ), черга ThreadPool порожня. Система працює значно нижче своєї межі — типовий патерн idle-like навантаження.

**Load-тестування (50 VUs, 5 хв).** Спостерігається виражена фаза прогріву: CPU зростає від початкових ~0,4% до 43% на середині тесту, з піком 80% у першій чверті — момент одночасного заповнення connection pool PostgreSQL та JIT-компіляції hot-path методів. Working Set зростає з 242 до 301 МБ. GC Heap демонструє пилкоподібний патерн (22→58→33→40→56 МБ) — типову поведінку Server GC з великим бюджетом пам'яті. ThreadPool стартує з 16 потоків (початковий burst), потім стабілізується на 11 — .NET runtime адаптує розмір пулу до усталеного навантаження. Черга ThreadPool короткочасно досягає 16, але швидко дренується.

**Stress-тестування (10→200 VUs, 10 хв).** Динаміка CPU відображає профіль ramping: 0,2% → 16% (25% часу) → 21% (50%) → 35% (75%) → 1,6% (кінець, drain). Working Set коливається в діапазоні 214–280 МБ без стійкого зростання — ознака відсутності memory leak. Черга ThreadPool з'являється у другій половині тесту (1–4 елементи при 100–200 VUs), але не накопичується — система встигає обробляти запити, хоча наближається до межі.

**Ultra-stress-тестування (50→1000 VUs, 15 хв).** Критичний сценарій з чіткими фазами:

Таблиця: Динаміка серверних метрик по фазах ultra-stress-тесту

| Фаза        | VUs      | CPU, % | Working Set, МБ | GC Heap, МБ | ThreadPool Queue |
| ----------- | -------- | ------ | --------------- | ----------- | ---------------- |
| Початок     | 50       | 0,1    | 246             | 35          | 0                |
| Прогрів     | 50→200   | 46,9   | 332             | 102         | 25               |
| Зростання   | 200→500  | 50,0   | 520             | 141         | 46               |

Ключове спостереження: CPU стабілізується на ~46–50% і не зростає далі навіть при 1000 VUs. Це підтверджує, що bottleneck — не обчислювальна потужність, а пропускна здатність I/O та управління конкурентністю. Черга ThreadPool зростає нелінійно: при 200 VUs = 25, при 500 VUs = 46, при 1000 VUs = 114 (пік 188). Working Set досягає 804 МБ (40% виділених 2 ГБ), GC Heap — 520 МБ. Після завершення навантаження система повністю відновлюється: Working Set повертається до 282 МБ, GC Heap — до 18 МБ, черга — до 0. Це свідчить про відсутність ресурсних витоків.

### 3.10 Виявлені bottlenecks

Під час тестування встановлено два рівні вузьких місць:

**1. Зовнішній — TMDB API rate limit.** При навантаженні до 200 VUs внутрішня інфраструктура має запас продуктивності. Обмежуючим фактором є rate limit TMDB API для ендпоінтів пошуку медіа.

**2. Внутрішній — ThreadPool saturation (1000 VUs).** Черга ThreadPool зросла до 188, p(95) до 3240 мс. CPU не досяг максимуму (avg 45,5%) — bottleneck пов'язаний з управлінням конкурентністю та GC pressure (heap 520 МБ, Working Set 804 МБ).

## 4 Висновки тестування продуктивності

Виконано чотири сценарії навантаження (smoke, load, stress, ultra-stress) для 15 ендпоінтів REST API з паралельним збором серверних метрик dotnet-counters.

Основні результати:

1. **Пороги якості дотримані** для smoke, load та stress. При робочому навантаженні (200 VUs) p(95) = 99,8 мс — значний запас до порогу 300 мс.

2. **Частка помилок 0%** на всіх рівнях — від 2 до 1000 VUs.

3. **Пропускна здатність** зростає лінійно: 13,3 → 56,4 → 149,9 → 527,2 req/s.

4. **Ultra-stress (1000 VUs)** — p(95) = 3240 мс > 1000 мс (PARTIAL FAIL), але 0% помилок. Bottleneck — ThreadPool queue (188) та GC pressure (heap 520 МБ).

5. **Ресурси** використовуються ефективно: при 50 VUs CPU avg = 27,8%, RAM = 261 МБ / 2 ГБ.
